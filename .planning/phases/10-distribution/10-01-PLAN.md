<tasks>

<task type="auto">
  <name>Task 1: Production build configuration fixes</name>
  <files>app.json, eas.json</files>
  <action>
1. Add `"expo-notifications"` to the plugins array in app.json (after "@react-native-community/datetimepicker"). This is required for notifications/alarms to work in production builds -- they work in Expo Go without it but the native module is not linked without the plugin entry.

2. Update eas.json submit.production to include iOS config with a placeholder ascAppId:
```json
"submit": {
  "production": {
    "ios": {
      "ascAppId": "REPLACE_WITH_APP_STORE_CONNECT_APPLE_ID"
    }
  }
}
```
The user will replace this placeholder after creating the App Store Connect listing in the checkpoint task.

3. Configure EAS secrets for Supabase environment variables so they are available during cloud builds. Run:
```bash
eas secret:create --scope project --name EXPO_PUBLIC_SUPABASE_URL --value "VALUE_FROM_LOCAL_ENV" --type string --force
eas secret:create --scope project --name EXPO_PUBLIC_SUPABASE_ANON_KEY --value "VALUE_FROM_LOCAL_ENV" --type string --force
```
Read the actual values from the local .env file to use in these commands. EXPO_PUBLIC_ prefixed vars are embedded at build time by Expo's Metro bundler -- they must be available as EAS secrets for cloud builds.

4. Run `npx jest --bail` to verify no regressions from config changes.
  </action>
  <verify>
    <automated>npx jest --bail</automated>
  </verify>
  <done>app.json has expo-notifications in plugins. eas.json has submit.production.ios with ascAppId placeholder. EAS secrets configured for Supabase env vars. Tests pass.</done>
</task>

<task type="auto">
  <name>Task 3: Trigger EAS Build with auto-submit to TestFlight</name>
  <files>eas.json</files>
  <action>
1. Verify eas.json has a real ascAppId (not the placeholder). If it still has the placeholder, abort and tell the user to complete the checkpoint.

2. Commit the updated eas.json with the real ascAppId.

3. Run the production build with auto-submit:
```bash
eas build --platform ios --profile production --auto-submit --non-interactive
```

Note: The first build will prompt for Apple credentials to generate managed signing certificates. If the --non-interactive flag causes auth issues, fall back to:
```bash
eas build --platform ios --profile production --auto-submit
```
and let the user authenticate interactively.

4. After the build starts, check status with:
```bash
eas build:list --platform ios --limit 3
```

The build takes 10-20 minutes on EAS servers. The --auto-submit flag will automatically upload the completed build to App Store Connect / TestFlight.

5. Once the build completes and submits, verify with:
```bash
eas submit:list --platform ios --limit 3
```

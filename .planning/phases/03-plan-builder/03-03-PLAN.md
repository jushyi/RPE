    - deletePlan: removes plan from Supabase (cascade deletes children) and store
    - setActivePlan: marks one plan active and deactivates others in store
    - Edit mode creates a deep clone draft; Cancel discards draft without persisting
  </behavior>
  <action>
    1. Extend `src/features/plans/hooks/usePlanDetail.ts`:
       - Add updatePlan(draft: Plan) function that:
         a. Updates workout_plans row (name, updated_at)
         b. Deletes existing plan_days for this plan (simpler than diffing)
         c. Inserts new plan_days from draft with sort_order = array index
         d. Inserts new plan_day_exercises for each day from draft with sort_order = array index within each day
         e. Updates planStore with the saved plan
         f. Returns success/error
       - This "delete-and-reinsert" approach for days/exercises is simpler than tracking individual adds/removes/edits for a v1 template editor

    2. Update `app/(app)/plans/[id].tsx` to add edit mode:
       - Add isEditing state (boolean, default false)
       - "Edit" button in header toggles isEditing=true
       - When entering edit mode: deep clone plan into local `draft` state (JSON parse/stringify for deep clone)
       - In edit mode:
         a. Plan name becomes editable Input
         b. Day sections become editable (reuse DaySlotEditor from Plan 02 create screen)
         c. Exercise rows become editable (PlanExerciseRow with isEditing=true)
         d. Can add/remove days, add/remove exercises, modify sets
         e. Header shows "Save" and "Cancel" buttons instead of "Edit"
       - Save button: calls usePlanDetail.updatePlan(draft), exits edit mode, refetches
       - Cancel button: discards draft, exits edit mode (no Supabase calls)
       - CRITICAL: Edit mode works on the deep-cloned draft -- never mutates the store directly (per RESEARCH.md pitfall 3)

    3. Add delete functionality:
       - "Delete Plan" button at bottom of plan detail screen (visible in edit mode)
       - Uses Alert.alert for confirmation: "Delete '[Plan Name]'? Past workouts logged with this plan will be kept."
       - Two buttons: "Cancel" (default) and "Delete" (destructive style)
       - On confirm: calls usePlans().deletePlan(id), navigates back to plan list with router.back()
       - Wire PlanCard swipe-to-delete: update PlanCard.tsx to replace the stub onDelete handler with a real handler that shows the same Alert.alert confirmation dialog, then calls usePlans().deletePlan(id)

    4. Wire active plan toggle:
       - On plan list: long-press PlanCard calls usePlans().setActivePlan(id)
       - On plan detail: add "Set as Active" button (visible when plan is not already active)
       - After setting active: show brief feedback (e.g., the active badge appears)

    5. Write tests in `tests/plans/plan-crud.test.ts`:
       - Test that edit mode creates a deep clone (modifying draft doesn't affect original)
       - Test deletePlan removes from store
       - Test setActivePlan marks correct plan active and others inactive
       - Mock Supabase calls

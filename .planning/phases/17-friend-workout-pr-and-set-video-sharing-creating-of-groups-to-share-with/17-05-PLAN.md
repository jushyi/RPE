    1. Create `src/features/social/utils/notifyGroup.ts`:
       - notifyGroupOnShare(groupId: string, sharerId: string, sharerName: string, contentType: 'workout' | 'pr' | 'video'): Promise<void>
       - Fetches non-muted group members (excluding sharer) via socialStore or direct Supabase query
       - Calls existing send-push Edge Function with recipient_ids, title, body
       - Title format: "{sharerName} shared a {contentType}" (e.g., "John shared a workout")
       - Body varies by type: workout -> "Check out their session", pr -> "New personal record!", video -> "Watch their set"
       - data payload includes: { type: 'group_share', group_id: groupId }
       - Fire-and-forget (catch errors with console.warn, don't block share flow)

    2. Create `src/features/social/hooks/useShareFlow.ts`:
       - Takes session (WorkoutSession), prs (array of { name, weight, unit })
       - Derives shareable content from session:
         - workoutSummary: always available (built from session)
         - prItems: one per PR exercise (from prs array)
         - videoItems: sets that have video_url (check logged_sets for video_url presence). Per Pitfall 6 from RESEARCH.md: only include videos with confirmed video_url, skip those still uploading
       - State: selectedContent (Set<'workout' | 'pr-{index}' | 'video-{index}'>), selectedGroups (Set<string>), sharing (boolean), shared (boolean)
       - Actions: toggleContent(key), toggleGroup(groupId), share()
       - share(): for each selected group, builds SharedItem objects using sharePayload utils, calls socialStore.shareToGroups, then calls notifyGroupOnShare for each group. Sets shared=true on success.
       - CRITICAL: Body metrics are NEVER included in any payload (SOCL-14). The share flow only accesses exercises, sets, PRs, and videos from the WorkoutSession.
  </action>
